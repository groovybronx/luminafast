# M0.1 - Analyse Root Cause

Statut : Completee
Date : 2026-03-12
Branche : phase/m0.1-cadrage-root-cause

## Symptome observe

Le projet maintient deux implementations Rust du traitement pixel en parallele (backend et WASM). Ces deux versions ont deja diverge dans leur logique et leurs capacites, ce qui introduit un risque de rendu incoherent entre preview et export.

## Cause racine technique

La logique metier image n'est pas centralisee dans une crate partagee. Le mode "copie manuelle" entre modules cree un couplage faible et aucune garantie de synchronisation lors des evolutions.

## Correction structurelle retenue

Creer une source de verite unique dans une crate Rust partagee (`luminafast-image-core`) et faire dependre a la fois `luminafast-wasm` et `src-tauri` de ce noyau. Les wrappers WASM et Tauri restent minces et ne portent plus de logique algorithmique dupliquee.

## Cadrage IN / OUT de M0.1

IN:

- Documenter symptome, cause racine, correction structurelle.
- Figer la decision d'architecture cible.
- Delimiter clairement le scope de migration.

OUT:

- Aucun refactor code production.
- Aucun changement API frontend/backend.
- Aucune optimisation algorithmique.

## Decisions

1. Architecture cible validee : crate partagee core image.
2. Execution par sous-phases M0 -> M5 via briefs dedies.
3. Validation systematique par gates avant phase suivante.

## Preuves techniques utilisees

- `src-tauri/Cargo.toml` declare la crate backend `luminafast`.
- `luminafast-wasm/Cargo.toml` declare la crate `luminafast-wasm`.
- `src-tauri/src/services/image_processing.rs` applique les filtres en passes multiples.
- `luminafast-wasm/src/image_processing.rs` applique une passe unique et expose histogramme.
